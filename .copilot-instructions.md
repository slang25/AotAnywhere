# GitHub Copilot Development Instructions

This document provides development guidance for the AotAnywhere repository, specifically focused on helping with agent tasks and iterating on the clang shim.

## Repository Overview

AotAnywhere is a NuGet package that enables cross-compilation for .NET Native AOT applications using Zig as the linker/sysroot. The primary components developers work with are:

- `src/clang_shim.zig` - Zig source for the clang wrapper. The MSBuild targets compile it on demand with the bundled zig toolchain (`CompileClangShim` target), so the same shim runs on every host OS. It rewrites clang arguments and forwards the invocation to `zig cc`. The source must compile with the zig version pinned by `$(ZigVersion)` in `Crosscompile.targets`; bumping that version may require updating the shim for zig std library changes.
- `src/link_shim.zig` - the `link` personality of the same multi-call binary, used when targeting Windows (`win-x64`, `win-arm64`) from a Linux/macOS host. It expands the MSVC-style `link @rsp` response file the ILC targets write, translates the options to GNU spellings, injects a glue C source bridging the MSVC-built NativeAOT runtime libs onto the MinGW CRT (GS-cookie helpers, MSVC-mangled operator new/delete, arm64 `_Interlocked*`, `_fltused`, `__isa_available`, empty stub archives for `LIBCMT`/`OLDNAMES`/`libcpmt`/`uuid`), and forwards to `zig cc -target <arch>-windows-gnu`.
- `src/Crosscompile.targets` - MSBuild targets that configure Zig toolchain and compile the shim
- `src/StuDev.AotAnywhere.targets` - Main MSBuild integration
- `src/apple-sysroot/` - Minimal, self-generated Apple linker stubs (`.tbd`) used when cross-compiling to macOS from Windows/Linux. Regenerated with `eng/generate-apple-sysroot.cs` (requires macOS with Xcode and zig on PATH); update `PackVersions` in that script when new .NET versions ship. `Crosscompile.targets` points the shim at it via the `AOTANYWHERE_APPLE_SYSROOT` environment variable unless the user supplies their own sysroot.

## Required Development Dependencies

### Zig Installation

Development and testing require the zig version pinned by `$(ZigVersion)` in `src/Crosscompile.targets` (currently `0.16.0.2`, i.e. **Zig 0.16.0**; the Vezel.Zig.Toolsets package version is the zig version plus a packaging suffix).

Install from [ziglang.org/download](https://ziglang.org/download/) (pick the archive matching your OS/arch and put the extracted directory on PATH), or use a package manager (`brew install zig`, `winget install zig.zig`, ...) if it carries the pinned version.

#### Verification

```bash
zig version
# Should output: 0.16.0
```

### Optional Dependencies

- **LLVM**: For symbol stripping via `llvm-objcopy`. Download from [LLVM releases](https://releases.llvm.org/)
- **.NET 8.0 SDK**: Required for building and testing the NuGet package

## Development Workflow

### Building the Project

```bash
# Build the NuGet package
dotnet pack src/AotAnywhere.nuproj -c Release

# Test with the sample Hello World project
dotnet build test/Hello.csproj

# Test cross-compilation (requires Zig)
dotnet publish test/Hello.csproj -r linux-x64
```

### Testing the Clang Shim

The clang shim (`src/clang_shim.zig`) is the core of cross-compilation functionality. The argument-filtering logic (including the macOS cross-compilation paths that are unreachable on a macOS host) is covered by unit tests in the same file:

```bash
# Run the unit tests (requires the zig version pinned in Crosscompile.targets)
zig test src/clang_shim.zig
```

For manual end-to-end iteration:

```bash
# Compile the shim for the host
zig build-exe -O ReleaseSafe -femit-bin=/tmp/clang src/clang_shim.zig

# Test basic zig availability check
/tmp/clang --version

# Test with sample compilation arguments (Linux target)
/tmp/clang -o test_output -target x86_64-linux-gnu test.c

# Test macOS cross-compilation detection
/tmp/clang -o test_output -target x86_64-apple-darwin test.c

# Inspect the argument rewriting without reading zig output:
ZIG_SHIM_DEBUG=1 /tmp/clang -o test_output -target x86_64-linux-gnu test.c
```

To verify the final `zig cc` invocation in isolation, put a fake `zig` first on the PATH that prints its arguments:

```bash
mkdir -p /tmp/fakebin
printf '#!/bin/sh\nprintf "ARG:%%s\\n" "$@"\n' > /tmp/fakebin/zig
chmod +x /tmp/fakebin/zig
PATH="/tmp/fakebin:$PATH" /tmp/clang -o hello --discard-all -lz main.o
```

### Common Development Tasks

#### Modifying Cross-Compilation Logic

When modifying the clang shim, focus on these key areas:

1. **Target Detection**: Logic that determines if compiling for Linux or macOS
2. **Argument Filtering**: Removing unsupported flags for Zig
3. **Library Handling**: Managing system libraries and frameworks
4. **Path Configuration**: Setting up SDK paths and search directories

#### Testing Changes

```bash
# Test all supported target platforms from repository root
dotnet publish test/Hello.csproj -r linux-x64
dotnet publish test/Hello.csproj -r linux-arm64
dotnet publish test/Hello.csproj -r linux-arm       # armv7; publishes as net9.0
dotnet publish test/Hello.csproj -r linux-musl-x64
dotnet publish test/Hello.csproj -r linux-musl-arm64
dotnet publish test/Hello.csproj -r linux-musl-arm  # armv7; publishes as net9.0
dotnet publish test/Hello.csproj -r osx-x64     # Experimental
dotnet publish test/Hello.csproj -r osx-arm64   # Experimental
dotnet publish test/Hello.csproj -r win-x64     # From Linux/macOS hosts (zig/MinGW link)
dotnet publish test/Hello.csproj -r win-arm64   # From Linux/macOS hosts (zig/MinGW link)
```

#### Debugging Cross-Compilation Issues

1. **Enable Verbose Output**: Set `ZIG_SHIM_DEBUG=1` to print original and final arguments
2. **Check Zig Command**: Review the final `zig cc` command being executed
3. **Validate Arguments**: Ensure unsupported flags are properly filtered
4. **Test Incrementally**: Make small changes and test after each modification

### Using External Zig vs. NuGet Package

The repository supports two modes for Zig usage:

#### Default: NuGet Package (Recommended for CI/CD)
```bash
# Uses Vezel.Zig.Toolsets package version pinned in Crosscompile.targets
dotnet publish test/Hello.csproj -r linux-x64
```

#### External Zig (Recommended for Development)
```bash
# Uses system-installed Zig from PATH
dotnet publish test/Hello.csproj -r linux-x64 /p:UseExternalZig=true
```

For development and iteration on the clang shim, using external Zig is preferred as it:
- Provides faster iteration cycles
- Allows easier debugging with verbose output
- Enables testing with different Zig versions

## Troubleshooting

### Common Issues

#### "zig is not on the PATH"
- Verify the pinned zig version is installed and in PATH
- Check with `zig version` command
- Restart terminal/IDE after PATH changes

#### Cross-compilation failures
- Check that target platform flags are correctly detected
- Verify system libraries are available
- Review filtered arguments in script output

#### Windows cross-compilation notes

- Only reachable from non-Windows hosts (on Windows the SDK's native MSVC link applies and the package is inert)
- The link shim writes its glue (`aotanywhere-msvc-glue.c`) and stub archives (`aotanywhere-msvc-stub-libs/`) next to the `link.rsp` in the native intermediate directory; `ZIG_SHIM_DEBUG=1` prints the original and translated argument lists
- New undefined MSVC CRT symbols on future .NET versions belong in `glue_source` in `src/link_shim.zig`; keep definitions exactly ABI-faithful (the arm64 `__security_push_cookie`/`__security_pop_cookie` follow MSVC's `crt/arm64/secpushpop.asm` contract)
- `/MERGE`, `/guard:cf` and `/CETCOMPAT` are intentionally dropped: zig cc cannot express them and they are not needed for correctness

#### macOS cross-compilation notes
- Links against the bundled `.tbd` stubs in `src/apple-sysroot`; unresolved Apple symbols on newer .NET versions mean the stubs need regenerating
- The shim injects a `aotanywhere-gs-pad.c` file into every macOS link: zig's linker keeps `__DATA,__const` next to `__DATA,__data`, and .NET's `InitGSCookie()` write-protects the GS cookie's whole page at startup - without the 16 KiB alignment pad the binary dies with SIGBUS in `InitializeGlobalTablesForModule`
- The shim also passes the Swift overlay libraries (`-lswiftCoreFoundation`, ...) explicitly plus `-Wl,-dead_strip_dylibs`, because zig's linker ignores the `LC_LINKER_OPTION` autolink commands in .NET's Swift crypto bindings
- `StripSymbols` defaults to `false` for osx RIDs (Apple strip/dsymutil reject zig-linked binaries and don't exist on other hosts)
- Binaries are ad-hoc signed by zig; sign/notarize on a Mac for distribution

## Code Style Guidelines

When modifying the clang shim:

1. **Keep it compatible with the pinned zig** - the shim is compiled by the zig version in `$(ZigVersion)` (and whatever users bring via `UseExternalZig`), and zig's std library changes between releases; run `zig test src/clang_shim.zig` after any change and whenever `$(ZigVersion)` is bumped
2. **Add comprehensive comments** for complex argument filtering logic
3. **Keep the unit tests in `clang_shim.zig` passing and extend them** when adding new argument rewrites
4. **Preserve existing error handling** patterns (forward `zig cc` exit codes, fail with a clear message when zig is missing)
5. **Test edge cases** like arguments containing spaces or special characters

## Performance Considerations

- Zig compilation is generally fast, but cross-compilation may take longer
- Consider using external Zig for development to avoid NuGet package download overhead
- Large applications with many dependencies may have longer link times
