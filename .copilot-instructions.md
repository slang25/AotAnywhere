# GitHub Copilot Development Instructions

This document provides development guidance for the AotAnywhere repository, specifically focused on helping with agent tasks and iterating on the cross-link tasks and MSBuild targets.

## Repository Overview

AotAnywhere is a NuGet package that enables cross-compilation for .NET Native AOT applications using Zig as the linker/sysroot. There is **no native shim binary** — every link is either an MSBuild takeover that runs `zig cc` directly, or a managed MSBuild task. The primary components developers work with are:

- `src/DirectLink.targets` - the MSBuild link takeovers. `AotAnywhereDirectLinkNative` (Linux) and `AotAnywhereDirectLinkMacNative` (macOS) run `BeforeTargets="LinkNative"`: they reconstruct the command line the ILC SDK's `LinkNative` would build, apply the zig-specific fixups, and `Exec` `zig cc` by absolute path. This file also wires the Windows link and the ELF strip to their tasks.
- `src/tasks/AotAnywhere.Tasks/` - the managed task assembly (netstandard2.0, one portable DLL shipped under `build/tasks/`). It contains: `MsvcArgTranslator` + `CoffSectionRenamer` + `AotAnywhereWindowsLink` (translate the MSVC `link.exe` args the ILC targets produce into a `zig cc -target <arch>-windows-gnu` MinGW cross-link, honour `/MERGE` by renaming COFF sections, and supply the glue + stub archives); and `ElfStripper` + `AotAnywhereStrip` (the Linux symbol strip — ELF surgery zig cannot do). The pure classes are unit-tested from `tests/AotAnywhere.MSBuild.Tests`.
- `src/aotanywhere-msvc-glue.c` - MSVC↔MinGW CRT glue (GS-cookie helpers, MSVC-mangled operator new/delete, arm64 `_Interlocked*`, `_fltused`, `__isa_available`) compiled into every Windows cross-link by `AotAnywhereWindowsLink`.
- `src/aotanywhere-gs-pad.c` - the 16 KiB GS-cookie page pad compiled into every macOS link by `AotAnywhereDirectLinkMacNative`.
- `src/Crosscompile.targets` - MSBuild targets that resolve the Zig toolchain (`SetPathToZig`) and point the ILC SDK's linker/objcopy probes at the real zig (`PointLinkerToZig`).
- `src/StuDev.AotAnywhere.targets` - Main MSBuild integration
- `src/Sdk/Sdk.props` - MSBuild SDK entry point (`<Sdk Name="StuDev.AotAnywhere" />`), the recommended way to consume the package. It injects the package and host Zig toolset references where NuGet restore can see them — restore does not evaluate PackageReferences declared inside a package's build/ targets (NuGet/Home#4790), so a plain PackageReference alone cannot fetch Zig on a clean machine. Both the package version and the `ZigVersion` default are baked in here at pack time (`BakeSdkProps` in the nuproj) — the latter from the single `ZigVersion.props` pin, so this file carries no version literal of its own (see `docs/zig-version-bump.md`).
- `src/apple-sysroot/` - Minimal, self-generated Apple linker stubs (`.tbd`) used when cross-compiling to macOS from Windows/Linux. Regenerated with `eng/generate-apple-sysroot.cs` (requires macOS with Xcode and zig on PATH); update `PackVersions` in that script when new .NET versions ship. The macOS link takeover reads `$(AotAnywhereAppleSysroot)` (defaulting to this directory) unless the user supplies their own sysroot.

## Required Development Dependencies

### Zig Installation

Development and testing require the zig version pinned by `$(ZigVersion)` in `src/ZigVersion.props` (the single source of truth; currently `0.16.0.2`, i.e. **Zig 0.16.0**; the Vezel.Zig.Toolsets package version is the zig version plus a packaging suffix). This prose pin is checked against `ZigVersion.props` by the `zig-version-sync` CI guard.

Install from [ziglang.org/download](https://ziglang.org/download/) (pick the archive matching your OS/arch and put the extracted directory on PATH), or use a package manager (`brew install zig`, `winget install zig.zig`, ...) if it carries the pinned version.

#### Verification

```bash
zig version
# Should output: 0.16.0
```

### Optional Dependencies

- **.NET 8.0 SDK**: Required for building and testing the NuGet package. (No LLVM/clang needed — the ELF strip is done in managed code and links go through `zig cc`.)

## Development Workflow

### Building the Project

```bash
# Build the NuGet package
dotnet pack src/AotAnywhere.nuproj -c Release

# Test with the sample Hello World project
dotnet build test/Hello.csproj

# Test cross-compilation (requires Zig)
dotnet publish test/Hello.csproj -r linux-x64
```

### Testing the link tasks

The MSVC→GNU translation, the `/MERGE` COFF surgery and the ELF strip are pure C# classes in `src/tasks/AotAnywhere.Tasks`, unit-tested (via the Microsoft.Build APIs and known-answer fixtures) in `tests/AotAnywhere.MSBuild.Tests`:

```bash
# Run the C# unit tests (translator, COFF renamer, ELF strip CRC, MSBuild logic)
dotnet run --project tests/AotAnywhere.MSBuild.Tests/AotAnywhere.MSBuild.Tests.csproj -c Release
```

For end-to-end iteration, publish a target and inspect the `zig cc` command line, which appears in the build output (and in a binlog via `-bl`):

```bash
dotnet publish test/Hello.csproj -r linux-x64  -c Release   # DirectLink takeover
dotnet publish test/Hello.csproj -r osx-arm64  -c Release   # DirectLink takeover (macOS)
dotnet publish test/Hello.csproj -r win-x64    -c Release   # AotAnywhereWindowsLink task
```

When iterating on the managed tasks in a source build, build the task assembly first so the targets resolve it from its bin output (the dev fallback in `DirectLink.targets`):

```bash
dotnet build src/tasks/AotAnywhere.Tasks/AotAnywhere.Tasks.csproj -c Release
```

### Common Development Tasks

#### Modifying the link logic

- **Linux/macOS link fixups** live in `DirectLink.targets` (the `_AotAnywhere*LinkArg` item groups): which SDK args to drop, and which zig-specific ones to add.
- **Windows MSVC→GNU translation and `/MERGE`** live in `MsvcArgTranslator` / `CoffSectionRenamer`; add a test in `tests/AotAnywhere.MSBuild.Tests` for any new option or rewrite.
- **ELF strip** lives in `ElfStripper`; it is validated against real binaries by the CI `validate` job (which runs the stripped output).

#### Testing changes

```bash
# Test all supported target platforms from repository root
dotnet publish test/Hello.csproj -r linux-x64
dotnet publish test/Hello.csproj -r linux-arm64
dotnet publish test/Hello.csproj -r linux-arm       # armv7; publishes as net9.0
dotnet publish test/Hello.csproj -r linux-musl-x64
dotnet publish test/Hello.csproj -r linux-musl-arm64
dotnet publish test/Hello.csproj -r linux-musl-arm  # armv7; publishes as net9.0
dotnet publish test/Hello.csproj -r osx-x64
dotnet publish test/Hello.csproj -r osx-arm64
dotnet publish test/Hello.csproj -r win-x64     # From Linux/macOS hosts (zig/MinGW link)
dotnet publish test/Hello.csproj -r win-arm64   # From Linux/macOS hosts (zig/MinGW link)
```

#### Debugging cross-compilation issues

1. **Read the `zig cc` command line** printed by the takeover (or the task's logged invocation) — it is the actual link.
2. **Capture a binlog** with `-bl` and open it to see the computed `@(LinkerArg)` and the reconstructed command.
3. **Validate arguments**: ensure unsupported flags are dropped and the target triple is present.
4. **Test incrementally**: make small changes and test after each modification.

### Using External Zig vs. NuGet Package

The repository supports two modes for Zig usage:

#### Default: NuGet Package (Recommended for CI/CD)
```bash
# Uses the Vezel.Zig.Toolsets package version pinned in ZigVersion.props
dotnet publish test/Hello.csproj -r linux-x64
```

#### External Zig (Recommended for Development)
```bash
# Uses system-installed Zig from PATH
dotnet publish test/Hello.csproj -r linux-x64 /p:UseExternalZig=true
```

For iterating on the link logic, external Zig is preferred: faster cycles, easier to swap zig versions.

## Troubleshooting

### Common Issues

#### "zig is not on the PATH"
- Verify the pinned zig version is installed and in PATH (or the Vezel toolset restored)
- Check with `zig version`
- Restart terminal/IDE after PATH changes

#### Cross-compilation failures
- Check the target triple is injected and correct
- Verify system libraries / runtime packs restored
- Review the reconstructed link command (or a binlog)

#### Windows cross-compilation notes

- Only reachable from non-Windows hosts (on Windows the SDK's native MSVC link applies and the package is inert)
- The `AotAnywhereWindowsLink` task writes its glue (`aotanywhere-msvc-glue.c` is shipped and compiled in) and empty stub archives (`aotanywhere-msvc-stub-libs/`) into the native intermediate directory
- New undefined MSVC CRT symbols on future .NET versions belong in `src/aotanywhere-msvc-glue.c`; keep definitions exactly ABI-faithful (the arm64 `__security_push_cookie`/`__security_pop_cookie` follow MSVC's `crt/arm64/secpushpop.asm` contract)
- `/MERGE` is implemented by renaming the affected sections in *copies* of the input objects (never in place — bootstrapper.obj lives in the shared NuGet cache), which makes lld produce link.exe's merged layout; `/guard:cf` and `/CETCOMPAT` are dropped: zig cc cannot express them and they are not needed for correctness
- `/OPT:REF` and `/OPT:ICF` are honored by a second link pass: zig cc cannot pass COFF `/OPT` flags through (it always hands lld-link `-DEBUG`, whose default is `OPT:NOREF,NOICF`), so the task re-runs the lld-link invocation `zig cc -v` prints — via the `zig lld-link` subcommand — with the flags appended (~15% smaller binaries, issue #65). The replayed line is unquoted, so paths with spaces are routed through symlink aliases in a temp scratch dir first (`LinkPathAliaser`; zig echoes given paths verbatim, and zig's cache dirs are redirected via `ZIG_GLOBAL_CACHE_DIR`/`ZIG_LOCAL_CACHE_DIR`). An untrustworthy replay fails the link with a clear error rather than silently shipping the unoptimized binary — the only unfixable shape is a space in the output file *name* (post-link renaming would desync the embedded PDB reference); `AotAnywhereWindowsLinkOptimize=false` opts out (single pass, no `/OPT`)
- The `/GS` cookie is randomized by a `.CRT$XCAA` initializer in the glue (runs before any MSVC-built initializer, so no GS frame is live across the change)

#### macOS cross-compilation notes
- Links against the bundled `.tbd` stubs in `src/apple-sysroot`; unresolved Apple symbols on newer .NET versions mean the stubs need regenerating
- The macOS takeover compiles `src/aotanywhere-gs-pad.c` into every link: zig's linker keeps `__DATA,__const` next to `__DATA,__data`, and .NET's `InitGSCookie()` write-protects the GS cookie's whole page at startup - without the 16 KiB alignment pad the binary dies with SIGBUS in `InitializeGlobalTablesForModule`
- It also passes the Swift overlay libraries (`-lswiftCoreFoundation`, ...) explicitly plus `-Wl,-dead_strip_dylibs`, because zig's linker ignores the `LC_LINKER_OPTION` autolink commands in .NET's Swift crypto bindings
- `StripSymbols` defaults to `true` for osx RIDs like everywhere else, honoured at link time via ld64's `-x`/`-S` (Apple strip/dsymutil reject zig-linked binaries and don't exist on other hosts, so there is no post-link strip and no dSYM sidecar)
- Binaries are ad-hoc signed by zig; sign/notarize on a Mac for distribution

## Code Style Guidelines

When modifying the link tasks or targets:

1. **Keep the SDK as the source of truth for _what_ to link** - reconstruct framing minimally and let `@(LinkerArg)` carry the bulk, so version drift lands in the drop list rather than the structure
2. **Add comprehensive comments** for non-obvious argument handling
3. **Extend the unit tests in `tests/AotAnywhere.MSBuild.Tests`** when adding a new translation, COFF/ELF behaviour, or MSBuild rewrite
4. **Preserve existing error handling** patterns (surface `zig cc` failures through MSBuild, fail with a clear message when zig is missing)
5. **Test edge cases** like paths containing spaces or special characters

## Performance Considerations

- Zig compilation is generally fast, but cross-compilation may take longer
- Consider using external Zig for development to avoid NuGet package download overhead
- Large applications with many dependencies may have longer link times
